# 1. Siltとは何か

Siltは、低レイヤーの事実を、できるだけ早い段階で型付きの形に引き上げようとする言語です。

そのために、次の四つを同じ設計空間に置きます。

- Lisp的なS式表面構文
- 依存型コア
- 数量付きの値使用
- 低レイヤーの表現制御

普通、これらは別々の世界にあります。Lisp的な言語は構文の柔軟性に強い。依存型言語は証明に強い。システムプログラミング言語は低レイヤー実装に強い。Siltの問いは、それらを一つの言語でどこまで正直に接続できるかです。

ここでいう「正直」とは、できないことを隠さないという意味だけではありません。値が実行時に残るのか、型検査だけに使われるのか。副作用がどの順番で許されるのか。layoutがどのバイト列になるのか。そうした境界を、なるべくプログラムの表面に出すという意味です。

最小のSiltプログラムはこう見えます。

```lisp
(claim id (Pi ((A 0 Type) (x A)) A))
(def id
  (fn ((A 0 Type) (x A)) x))
```

`claim` は名前の型を宣言します。`def` は実装です。`Pi` は依存関数型です。`A 0 Type` は、`A` が型引数としてだけ使われ、実行時の値としては使われないことを示します。

この小さな例に、Siltの基本姿勢が入っています。型を先に読む。値がどこで使われるかを見る。実行時に残るものと、検査だけに使うものを分ける。

Siltの低レイヤー側は、次のように書きます。

```lisp
(layout BootState 16 8
  ((phase U64 0)
   (code U64 8)))

(static-cell limine-boot-state BootState)
```

これは単なるレコード定義ではありません。`BootState` はサイズ16、アラインメント8を持つ実行時レイアウトです。`phase` はオフセット0、`code` はオフセット8に置かれます。`static-cell` はこの型のゼロ初期化された静的領域を生成します。

さらにSiltは、x86のI/Oポート操作も型付きeffectとして表現します。

```lisp
(claim qemu-debug-exit
  (Pi ((pre 0 Type)
       (post 0 Type)
       (code U64))
      (Eff pre post Unit)))
```

ここで `Eff pre post Unit` は「状態 `pre` から `post` へ進み、値としては `Unit` を返す」操作です。

Siltは、完成した一般用途言語ではなく、低レイヤー依存型言語としての実装を積み上げている研究言語です。それでも、単なる構文実験ではありません。Siltはfreestanding Cを生成し、x86_64 ELFとしてリンクし、Limineで起動し、QEMU上でSilt由来のシリアル出力とdebug-exitを観測できます。

この本では、そこへ至る道筋を分解します。最初は小さな `claim` と `def` から始め、最後にはallocator handoffという低レイヤーの総合例まで読みます。
